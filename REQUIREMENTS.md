## 1. プロジェクト概要

### 1.1 目的
ビジネス思考フレームワーク（マトリクス図・ロジックツリー・ベン図など）を、閲覧するだけでなく**その場で編集・保存・書き出し**できる Web ツール集を提供する。一般公開を前提とする。

### 1.2 価値提案
既存の解説サイトや書籍は「読む」ことが中心だが、本ツールは各フレームワークを**操作可能なテンプレート**として提供し、ユーザー自身の課題を当てはめて思考を整理できる点に価値を置く。

### 1.3 対象ユーザー
- ビジネス上の意思決定・分析・計画・改善を行う個人および実務者
- 思考の型を手元で試しながら使いたい学習者

### 1.4 提供形態
- 静的 Web サイト（GitHub Pages 公開）
- ローカル保存（v1 は localStorage のみ。サーバー・アカウントは持たない）

---

## 2. スコープ

### 2.1 含むもの
- 6種の汎用エンジンで表現できるビジネス思考フレームワークの編集・保存・書き出し
- フレームワークの一覧・個別表示
- ブラウザローカルでのデータ永続化

### 2.2 含まないもの（現時点）
- サーバー連携・ユーザーアカウント・クラウド同期
- 条件分岐フローチャート等の業務手順図（後述「対象外」）
- 専用エンジンを必要とする特殊図（後述「将来拡張保留」）

## 3. 用語

| 用語 | 定義 |
|---|---|
| フレームワーク | 1つの思考ツール定義（data/frameworks.json の1要素）。`id` で一意。 |
| エンジン | 「型」ごとの汎用描画・編集ロジック（js/engines/ 配下）。 |
| 型 | 見た目・構造による分類。6種（matrix/tree/flow/sheet/venn/timeline）＋ sheet の grid2x2 レイアウト。 |
| カテゴリ | 内容ベースの7分類（戦略・分析 など）。型とは別軸。 |
| ロット | データ投入の作業単位（カテゴリ単位・数件ずつ）。 |

## 4. 技術スタック・アーキテクチャ

### 4.1 技術スタック
- HTML / CSS / バニラ JavaScript（外部フレームワーク不使用）
- データは `data/frameworks.json` で一元管理
- 保存はブラウザの localStorage（v1 はローカル保存のみ）
- メインカラー `#2563EB` / フォント Noto Sans JP

### 4.2 設計思想（データ駆動）
個別に85個を作り込むのではなく、**型ごとに1つの汎用エンジン**を用意し、`frameworks.json` のデータを流し込んで展開する。型の経済性を最優先し、1フレームワークのために専用エンジンを乱立させない。

### 4.3 エンジン構成（6種＋grid2x2）

| 型 | エンジン | 概要 |
|---|---|---|
| 型A マトリクス | `js/engines/matrix.js` | 2軸の4象限散布。項目をドラッグ配置（相対座標 0–1）。 |
| 型B ツリー/階層 | `js/engines/tree.js` | 再帰ノード。horizontal/vertical。SVG接続線。 |
| 型C プロセス/フロー | `js/engines/flow.js` | linear/cycle。SVG矢印。 |
| 型D 記入式シート | `js/engines/sheet.js` | row / stack（フィールド式）＋ grid2x2（4区分の箇条書き）。 |
| 型E 円/集合図 | `js/engines/venn.js` | 2集合/3集合。項目を自由配置（相対座標）。 |
| 型F 時系列/線表 | `js/engines/timeline.js` | 列スケール × タスクバー（grid-column）。 |

### 4.4 配線とページ
- `data/frameworks.json`（定義データ）→ `js/registry.js`（id解決・エンジン割当）→ 各ページ
- `index.html`：カテゴリ別の一覧・検索
- `view.html`：`?id=` で個別フレームワークを表示・編集
- `js/storage.js`：localStorage 読み書きの共通処理
- 注意：一覧・個別表示は `fetch()` を使うため `file://` では動作しない。GitHub Pages か `python -m http.server` 等の HTTP 配信が必要。

### 4.5 ファイル構成

## 5. データモデル（frameworks.json）

トップレベルは**フレームワーク定義オブジェクトの配列**。全要素が共通して `id`（一意）・`type`・`category`・`title` を持ち、`type` ごとに固有フィールドを持つ。

### 5.1 共通フィールド
| キー | 説明 |
|---|---|
| `id` | 一意の識別子（英小文字・ハイフン）。`view.html?id=` で参照。 |
| `type` | `matrix` / `tree` / `flow` / `sheet` / `venn` / `timeline` のいずれか。 |
| `category` | 7カテゴリのいずれか（内容ベース）。 |
| `title` | 表示名。 |

### 5.2 型別スキーマ

**matrix**
```json
{
  "axes": { "x": { "left": "...", "right": "..." }, "y": { "top": "...", "bottom": "..." } },
  "quadrants": { "topLeft": "...", "topRight": "...", "bottomLeft": "...", "bottomRight": "..." },
  "items": [ { "id": "...", "text": "...", "x": 0.0, "y": 0.0 } ]
}
```
`x`/`y` は 0–1 の相対座標（x: 左→右、y: 上→下）。

**tree**
```json
{
  "direction": "horizontal",
  "root": { "id": "...", "title": "...", "text": "...", "children": [ /* 同形のノードを再帰 */ ] }
}
```
ノードは `{id, title, text, children}`。`title`＝構造ラベル、`text`＝記入例。両者を混在させない。

**flow**
```json
{ "layout": "linear", "steps": [ { "id": "...", "title": "...", "text": "..." } ] }
```
`layout` は `linear` または `cycle`。

**sheet（row / stack）**
```json
{ "layout": "stack", "fields": [ { "id": "...", "label": "...", "value": "" } ] }
```
`row`＝ラベル左｜記入欄右の2カラム、`stack`＝ラベル上／記入欄下のカード。

**sheet（grid2x2）**
```json
{ "layout": "grid2x2", "cells": [ { "id": "tl", "label": "...", "items": [] } ] }
```
`cells` は `tl`/`tr`/`bl`/`br` の4要素固定（配列順は上段→下段）。

**venn**
```json
{ "sets": [ { "id": "A", "label": "..." } ], "items": [ { "id": "...", "text": "...", "x": 0.0, "y": 0.0 } ] }
```
`sets` は2または3。`items` は相対座標で自由配置。

**timeline**
```json
{ "scale": [ "Q1", "Q2", "..." ], "tasks": [ { "id": "...", "label": "...", "start": 0, "span": 1 } ] }
```
`scale` が列見出し、`tasks` は `start`（0始まりの列位置）と `span`（列数）で帯を配置。

## 8. 非機能要件・ハウスルール

| 区分 | ルール |
|---|---|
| 描画方式 | Canvas は不使用。CSS＋div で描画し、SVG は線（接続線・矢印）のみに使う。 |
| 座標系 | 線・座標を扱うエンジン（flow / tree）は `_canvasEl` 基準の座標系に統一。 |
| XSS対策 | ユーザー入力テキストは `textContent` のみで描画。`innerHTML` はクリア操作以外で使わない。 |
| 日本語入力 | Enter キー確定は `!e.isComposing` の IME ガードを必須とする。 |
| 保存 | v1 は localStorage のみ。 |
| 配信 | `fetch()` 利用のため HTTP 配信前提（file:// 不可）。 |

## 9. コンテンツ・著作権方針（厳守）

- 解説文・サンプル例文は、他サイト・書籍の文章を**一切参照・転載しない**。フレームワークの一般的な名前・構造のみを参考に、**完全オリジナル**で作成する。
- 収録数の目安（参考ページ上の約85種）は**クォータではない**。各候補は次の4条件を満たすこと。満たさなければ件数のために採用せず**見送る**。
  1. 既存6型＋grid2x2 に素直に乗る（無理に押し込まない）。
  2. 名前と一般構造のみ参照し、解説・例文は完全オリジナル。
  3. 思考ツールとして実質が成立する（枠だけの水増しにしない）。
  4. 既存フレームワークや他候補と機能的に重複しない。
- 各カテゴリは良質な候補が尽きたら打ち止め。少なすぎは不可だが、85に合わせるための質の低い収録は避ける。

## 10. エンジン拡張スコープ・保留/対象外

- 新規エンジンは原則追加しない（型は6＋grid2x2 で打ち止め）。
- **フィッシュボーン（特性要因図）**：将来拡張として保留。斜め骨の SVG 座標計算が難所のため、需要が確実になれば単独タスクで個別検討。
- **分岐フロー／フローチャート（条件分岐）**：収録対象外。本ツールは「思考フレームワーク」が主旨で、業務手順図は範囲外。
- **その他（散布図・ファネル・比較表など）**：投入時に都度判断。既存型で近似できるものはそれで扱う。なお**比較表は sheet エンジンが多列グリッド非対応のため見送り済み**（多列の選択肢×観点を表現できない）。

## 11. 開発・検証プロセス

### 11.1 体制
- **だいき**：オーナー兼開発者。Claude Code への指示を中継し、最終確認（スーパーリロード＋GitHub Pages）を行う。
- **PM/レビュー役（claude.ai）**：候補選定・JSON草案作成・機械検証・Claude Code 向け日本語プロンプト作成・プッシュ後の独立検証。
- **Claude Code**：実装担当。

### 11.2 ロットワークフロー（データ投入）
1. tarball で最新リポジトリのベースライン確認
2. JSON草案作成（型別スキーマ準拠）
3. 機械検証（id重複・必須キー・座標範囲・再帰ノード構造・カテゴリ集計）
4. だいきへ草案提示・承認
5. Claude Code 向け投入プロンプト作成（**push 指示を必ず含める**）
6. プッシュ後、tarball で end-state を独立検証（件数・新規id・既存無傷・JSON妥当）

### 11.3 検証の鉄則
- **報告を鵜呑みにせず、必ず tarball で実体を突き合わせる。**
  ```
  curl -sL "https://codeload.github.com/Daiki-Tk/logic-frameworks/tar.gz/refs/heads/master" -o repo.tgz && tar xzf repo.tgz
  ```
- `raw.githubusercontent.com` はプッシュ直後に古い CDN キャッシュを返すため検証に使わない。`codeload` の tarball はキャッシュ免疫。
- Git 運用：1コミット＝1論理単位、コミットメッセージは日本語。
